Result
From the Motors Analytics case

What this workflow saved on the core team

5–10×
less core-team capacity burned
Pre-AI plan (engineering lead estimate)

3–5 months · ~780–1,300 SP

  • Team: 1 BE + 1 FE + 1 QA building it from scratch
  • The engineering lead put together this team plan internally; the months are an industry baseline at about 70% focus on this project (real-life is usually longer).
What we actually did

38 days · 136 SP

  • Build: 1 PM + AI
  • Same core team: 1 BE + 1 FE + 1 QA reviewed & hardened
This is not about replacing devs. It's about shipping projects that otherwise wouldn't get done, and freeing the core team for the work only they can do.

How we got the numbers. The engineering lead's team plan for the analytics dashboard MVP is 1 BE + 1 FE + 1 QA. Time wasn't stated. Industry baseline for a standalone microservice, 3-page dashboard, 3-source integration (ClickHouse, Django, Redis), auth and GDPR is about 3–5 months at small-team pace, with room for rework, hand-offs and other priorities. At ~70% focus, the SP capacity is 3 people × 3–5 months × ~62 SP per person per 2-week sprint × 0.7 = ~780–1,300 SP. What we actually used across tracked tickets is 136 SP over 38 days. Ratio: 5.7–9.6×. Not counted: PM time, AI tools (~$600), pre-existing infra.

Stage 3
4

Pre-Dev: QA Check

QA validates the spec, not the code. All edge cases captured before development.

What QA checks

  • All states described (empty, error, edge)
  • AC clear, scope-out points explicit
  • Enough information for test design

In Motors

  • Ran spec through a structured task review with QA
  • 15 questions closed in one pass
  • All updates landed in spec before code started
Why: bugs are caught in writing, not in testing. QA doesn't burn time on issues that could have been prevented by a better spec.

Pre-Dev: Tech Review

CTO + backend lead + frontend lead validate architecture before coding.

What they review

  • Stack and integrations with existing services
  • API format, auth flow
  • Deploy strategy, infrastructure standards

In Motors

  • Proposed architecture → agreed with the backend lead
  • FastAPI + React, ClickHouse + Django Core
  • HttpOnly cookie + JWT via shared auth
Why: don't redo the architecture after the code is written. The lead suggested patterns I wouldn't have guessed, like JWT through a shared auth secret across services.

Stage 7
The key process change

Code Review: reviewer fixes the code,
no bounce-back to PM

before · earlier MR

FSD-lite refactor

Apr 14 to 21 · 7-day cycle
  1. PM creates MR (23 commits)
  2. Reviewer writes 4 issues in comments
  3. PM reads, makes 5 commits with fixes themselves
  4. PM writes “Done on all 4 items” with mapping
  5. MR merged
after · recent MR

v1.x bugfixes

May 8 to 18 · 10-day cycle
  1. PM creates MR + description
  2. Reviewer pushes 5 commits directly to the same branch: bug fixes, layered auth, redirect handling, priority tests
  3. 0 review comments bouncing back to PM
  4. MR merged straight to QA
Why: the reviewer already knows how to fix it. Looping through the PM on technical issues only adds delay. The PM only gets product questions (logic, spec, AC).

Advantages

What we got

01

Speed

38 days from custdev data to production. 2 days from data to prototype. Only 8 hours of live pair-call time with a developer. The rest is Claude Code and async review (full team review cost is about 174 person-hours across MVP and follow-ups, see deep dive).

02

PM stays focused on the product

One spec for everyone, so fewer rounds. The reviewer fixes technical comments themselves. PM doesn't waste time on "fix the react-router import".

03

This isn't about replacing the team. It's about making AI code safe to ship.

Designer, QA, reviewers, devops all stay in their roles. The spec discipline and the pre-dev gates are what let AI-written code clear the same review bar as anything else. The 1-PM build is the extreme case. The same workflow works with developers using AI as a pair-programmer, and the gates pay off the same way.

04

Designers + UX research adopt the tools too

The designer started using Claude Code, Wispr and Figma Make for her own work. One spec for everyone: designer, PM and dev all read the same document.

Drawbacks and open questions

Where the bottlenecks are right now

01

Dependency on core team capacity

3 weeks of Motors stuck on code review because core didn't have people. AI workflow speeds up dev, not review. The fix: reviewer pushes changes directly (see slide 13).

02

Spec quality is the bottleneck

If the PM can't write a clear spec with all edge cases, Claude Code will fill in the gaps with guesses. This needs disciplined writing. The quality bar is in the playbook (link at the end).

03

Best fit is greenfield, not legacy

The case study was a standalone microservice with a clean data domain. AI works much harder on legacy code with poor test coverage. Add a test harness first, then let AI touch the legacy parts.

04

Scaling across PMs is an open question

It worked with one PM with deep technical background. Can it be reproduced with 3-5 PMs? I don't know yet. A pilot is needed.

Requirements for participants

What each role needs to do

PM

  • One spec for everything: all edge cases, formulas and AC spelled out
  • Functional prototype, deployed
  • Self-test before handing off to review
  • Answer product questions from dev and QA
  • Don't touch code style or technical review

Developer / Reviewer

  • Be available for inline questions during Development
  • Help with MR flow, branches, CI
  • In Post-Dev Code Review: fix and refactor yourself
  • Only bounce product questions back to PM
  • Apply QA fixes in the same MR branch

QA

  • Validates the spec before coding (Pre-Dev: QA Check)
  • Flags edge cases the PM missed
  • In QA Testing: tests the final product
  • Bugs → back to Code Review, not to PM
